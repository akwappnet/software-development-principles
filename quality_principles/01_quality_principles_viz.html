Fail Fast
üè• Like Medical Checkups
It's better to find problems early (regular checkups) than wait until you're seriously ill. Early detection = easier treatment!
When something goes wrong in your code, make it fail immediately and clearly rather than continuing with bad data that causes bigger problems later.
Real Example:
Instead of accepting invalid email formats and failing silently, immediately show "Invalid email format" when user types wrong format.

Principle of Least Surprise
üöó Like Driving Any Car
No matter which car you drive, the brake pedal is always on the left, gas on the right. You don't have to relearn for each car!
Your software should work exactly how users expect it to work. Don't surprise them with unexpected behavior or unusual interfaces.
Real Example:
A "Save" button should save the document, not delete it. A red button typically means danger/delete, green means safe/proceed.

Convention over Configuration
üè† Like Building Houses
Most houses have doors at the front, light switches near doorways, and bathrooms upstairs. Following these conventions means less planning needed!
Use sensible defaults and standard patterns so developers don't have to configure everything from scratch. Make the common case simple.
Real Example:
A web framework assumes your database is called "database.db" and your homepage is "index.html" unless you specify otherwise.

Immutability
üìú Like Historical Documents
Once a historical document is written, you don't erase and change it. You create a new version with updates, keeping the original safe!
Instead of changing existing data, create new copies with your changes. This prevents bugs from unexpected modifications and makes your code more predictable.
Real Example:
Instead of modifying a user's profile directly, create a new profile object with the changes, leaving the original untouched.

Single Source of Truth
üìö Like a Library Catalog
A library has ONE master catalog that tells you where every book is. Imagine the chaos if different catalogs said different things!
Each piece of information should have exactly one authoritative source. Don't duplicate data across multiple places where they can get out of sync.
Real Example:
Store user's email in ONE database table, not scattered across multiple tables where they might become inconsistent.
