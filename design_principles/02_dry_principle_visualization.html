Don't Repeat Yourself
üè† Think of Your House

BAD: Having 5 different remote controls for the same TV

DRY Principle: Every piece of knowledge should have a single, unambiguous representation in your system

Problems:
  • Same validation logic copied 3 times
  • Bug fix needed in 3 places
  • Easy to forget updating all copies
  • Code bloat and maintenance nightmare
üåµ DRY Code (GOOD)
EmailValidator.java
public class EmailValidator {
    public static boolean isValid(String email) {
        if (email == null || email.isEmpty()) {
            showError("Email is required");
            return false;
        }
        if (!email.contains("@")) {
            showError("Invalid email format");
            return false;
        }
        return true;
    }
}
Usage everywhere
// UserRegistration.java
EmailValidator.isValid(email);

// UserLogin.java
EmailValidator.isValid(email);

// UserProfile.java
EmailValidator.isValid(email);
Benefits:
  • Single source of truth
  • Fix bug once, fixed everywhere
  • Consistent behavior across app
  • Less code, easier maintenance

üö® Common DRY Violations You've Probably Done

üî¢ Magic Numbers

Bad: Using 30, 60, 100 scattered throughout code

Good: Constants.TIMEOUT_SECONDS = 30

üìù Copy-Paste Code

Bad: Copying entire methods and changing 1 line

Good: Extract common logic into shared method

üé® Repeated Styles

Bad: Same color values repeated in multiple XML files

Good: colors.xml with named colors

üìÑ Duplicate Strings

Bad: "Loading..." text written 10 times

Good: strings.xml with @string/loading

üîÑ Repeated Logic

Bad: Same date formatting code everywhere

Good: DateUtils.formatDate() method

üèóÔ∏è Similar Classes

Bad: UserAdapter, ProductAdapter with 90% same code

Good: Generic BaseAdapter class
